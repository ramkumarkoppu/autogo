# Autonomous research loop

Iterate on a single experiment script, keeping improvements and reverting regressions.

User's research goal: $ARGUMENTS

## Two workflow tracks

Pick the one that matches the goal. The loop shape is identical; the script
you fork, the run command, and the metric differ.

### (a) Training tuning

Optimize how a checkpoint is trained: LR/schedule, batch size, steps,
weight decay, optimizer, augmentation, loss formulation, model config.

- Fork `experiments/2026-04-26_22-32-train-fromscratch/train.py`.
- Each run trains one iteration on a fixed dataset and emits a `===RESULT===`
  JSON line.
- **Default metric**: `train_policy_acc` (higher = better) or `train_loss`
  (lower = better). The reference `train.py` does NOT compute val metrics;
  to optimize on `val_loss` you must add a held-out val manifest + eval
  pass to your forked script first.

### (b) Test-time compute tuning

Optimize MCTS / search hyperparams used during game generation: number of
simulations, PCR ratios, c_puct, temperature, leaf batch size, resign
threshold, dirichlet noise. The model checkpoint is held fixed.

- Fork `experiments/2026-04-26_22-32-train-fromscratch/run_games.py` into a
  head-to-head harness (`eval_search.py` below).
- Two MCTS agents share the same frozen checkpoint but use different
  hyperparams (variant vs baseline). Play N games per color so komi is
  symmetric.
- **Default metric**: `win_rate` (higher = better) — variant wins divided
  by total head-to-head games.

All keep/discard decisions use the chosen target metric, with direction
implied by the metric (loss = lower-better; acc / win_rate = higher-better).

## Setup

1. Propose a tag like `<MMMDD>-<goal>` (e.g. `apr18-mctstemp`). Branch
   `autoresearch/<tag>` must not exist.
2. `git checkout -b autoresearch/<tag>`
3. Create a flat experiment folder:
   ```
   experiments/<YYYY-MM-DD_HH-MM>-<slug>/
       train.py          # (a) — the only file you edit between runs
       eval_search.py    # (b) — the only file you edit between runs
       results.tsv       # Experiment log (tab-separated, untracked)
       README.md         # Goal and notes
       progress.png      # Generated by analysis
   ```
   Use `TZ="America/Los_Angeles" date +"%Y-%m-%d_%H-%M"` for the prefix.
   Just `mkdir -p` — no subdirectories.
4. Initialize `results.tsv` with the header row only.

## Forking guidance

### (a) Training fork

Copy `experiments/2026-04-26_22-32-train-fromscratch/train.py` and adjust:

- **Dataset**: pass `--dataset-txt experiments/<EXP>/dataset-train.txt`.
  Manifest lines are paths under `$GAME_DATA_DIR` (default
  `/nfs/game_data_root`); the existing `_find_dataset(manifest)` helper
  resolves them.
- **Time budget**: keep `TIME_BUDGET_SECONDS` constant across runs (600s
  is a good fast-loop default) so runs are comparable.
- **`===RESULT===` schema** the reference emits:
  `iteration`, `train_loss`, `train_policy_acc`, `train_value_acc`,
  `steps_completed`, `elapsed_seconds`, `peak_vram_mb`, `checkpoint`,
  `resumed_from`. Add `val_loss` / `val_policy_acc` / `val_value_acc` /
  `n_params` if you need them.

### (b) Test-time-compute fork

Build a head-to-head harness (`eval_search.py`) — sketch:

```python
# Frozen checkpoint, two MCTS configs, head-to-head N games / color.
import json, time, sys, numpy as np
from pathlib import Path
from alpha_go.agents.base import register_agent
from alpha_go.agents.nn_mcts import CppMCTSAgent, LeafBatchedNNEvaluator

CKPT = "/nfs/checkpoints/<src-exp>/iterN_best.pt"
N_GAMES_PER_COLOR = 64

# --- Variant (edit between runs) ---
VAR = dict(num_simulations=512, c_puct=0.5, temperature=0.3,
           pcr_sims=[512, 1024], pcr_probs=[0.95, 0.05],
           leaf_batch_size=8, add_noise=False)
# --- Baseline (frozen) ---
BASE = dict(num_simulations=1024, c_puct=0.5, temperature=0.3,
            pcr_sims=[1024, 2048], pcr_probs=[0.95, 0.05],
            leaf_batch_size=8, add_noise=False)

def make(name, hp):
    @register_agent(name)
    class _A(CppMCTSAgent):
        def __init__(self):
            super().__init__(
                evaluator=LeafBatchedNNEvaluator(CKPT, 9, "18M"), **hp,
                resign_threshold=0.05, resign_consec_turns=5,
            )
    from alpha_go import self_play
    self_play._AGENT_MODEL_CONFIGS[name] = "18M"
    return name

variant = make("variant", VAR)
base    = make("baseline", BASE)

# Run alpha_go.self_play twice (variant black, variant white), point each
# at a per-run scratch save-name, then count winner==1/2 from the NPZs.
# Print ===RESULT=== with win_rate, total_games, elapsed_seconds.
```

Write a per-run scratch dir under
`$GAME_DATA_DIR/experiments/<EXP>/eval-runN/` and tear it down between
runs (or version it by run number — the autoresearch loop only needs the
parsed `===RESULT===` line).

## What you edit

**Only the experiment folder's contents.** Inside the folder, everything
is fair game. You may import from `alpha_go.model`, `alpha_go.dataset`,
`alpha_go.self_play`, `alpha_go.agents.*` — but do NOT modify those files.

Forbidden:
- Modifying `src/` without asking
- Installing new packages
- Changing the eval / metric definition (it's the ground truth)
- Changing the time / game budget between runs (keep comparable)

## Running one experiment

Direct `uv run` is the simplest path; both stdout streams contain the
`===RESULT===` line.

```bash
# (a) Training
uv run experiments/<EXP>/train.py \
    --dataset-txt experiments/<EXP>/dataset-train.txt --iteration 0

# (b) Test-time compute
uv run experiments/<EXP>/eval_search.py
```

For long runs you can dispatch to a single GPU worker via
`infra.remote_exec` (`--role train` / `--role collect`); see
`experiments/2026-04-26_22-32-train-fromscratch/run_iteration.sh` for the
SSH+docker invocation. The autoresearch loop does NOT need cluster
parallelism — it's strictly sequential.

**Timeout**: if a run exceeds 2× the time / game budget, kill it and log
as crash.

## Logging results

Append one line per run to `results.tsv` (tab-separated):

```
metric	memory_gb	status	description
```

- `metric`: the optimization target value (e.g. `train_loss`,
  `train_policy_acc`, `win_rate`). Single column — direction is implied
  by the metric name in the header.
- `memory_gb`: peak VRAM rounded to .1f (`peak_vram_mb / 1024`; 0.0 on
  crash).
- `status`: `keep`, `discard`, or `crash`.
- `description`: short text of what this run tried.

Example (training, lower is better):
```
train_loss	memory_gb	status	description
2.991546	0.3	keep	baseline: 256ch x 14b, lr=1.5e-3
2.937570	0.3	keep	lr=3e-3
2.725063	0.5	discard	bs=256, channels=320 — only 5303 steps
0.000000	0.0	crash	lr=3e-2 — diverged
```

Example (test-time compute, higher is better):
```
win_rate	memory_gb	status	description
0.500	0.4	keep	baseline: sims=1024 c_puct=0.5
0.580	0.4	keep	sims=2048
0.520	0.4	discard	temperature=0.5 (regressed)
```

Do NOT commit `results.tsv`. Add to `.gitignore` if missing.

## The loop

LOOP FOREVER:

1. Read `results.tsv` for the running best and current edit-target script.
2. Pick one change to try. Modify the script.
3. `git commit` with a short description of the change.
4. Run the script (see above).
5. Parse the `===RESULT===` line from stdout / log.
6. On crash (no result, missing metric): check the log for the error.
   Simple fix (typo, import) → fix and re-run. Otherwise log as crash and
   move on.
7. Append to `results.tsv`.
8. If target metric improved (lower for losses, higher for accs / win
   rates): keep the commit.
9. Else: revert the script:
   ```bash
   git checkout HEAD~1 -- experiments/$EXP/<script>.py
   git commit -m "revert: <description>"
   ```

### Strategy

- **Simpler beats complex at equal metric.** Removing code for no
  regression = win.
- **First run = baseline.** Run the unmodified script.
- **One variable at a time** when possible.
- **Training tuning**: LR + schedule are usually highest-leverage, then
  batch size and step count, then optimizer / augmentation / architecture.
- **Test-time compute tuning**: number of simulations is usually
  highest-leverage; PCR ratio second; temperature / c_puct often noise
  unless the model is very over- or under-confident. Pair noise-prone
  metrics with enough games (≥ 128 total) to avoid chasing variance.
- **Stuck?** Re-read the script fresh, combine near-misses, try radical
  changes (different optimizer, architecture tweak, MCTS variant). Revert
  if worse. Step back: are we investigating the right bottleneck?

## NEVER STOP

Do not pause to ask "should I keep going?". Run until manually
interrupted. ~12 min per training run ⇒ ~5/hr ⇒ ~40 overnight. Test-time-
compute runs are typically faster (no training step).

## Analysis

### Progress plot (`progress.png`)

Target metric (y) vs experiment number (x):
- Discarded runs: faint gray dots
- Kept runs: green dots with black edges
- Running best line (improvement frontier; `min` for losses, `max` for
  win rates / accs)
- Annotate kept runs with their description (rotated 30°)
- Y-axis zoomed to [best, baseline] × 1.15 (× 0.85 if higher-is-better)

### Summary

Print to console and append to `report.md`:
- Baseline vs best metric (absolute and %)
- Run counts: keep / discard / crash
- Keep rate
- Largest single-run improvement

### Top improvements

Rank kept runs by delta vs previous kept run. Surfaces which changes
mattered most.

## Datasets

Manifest files (e.g. `experiments/<exp>/dataset-itN.txt`) list paths
relative to `$GAME_DATA_DIR` (defaults to `/nfs/game_data_root`).
`train.py::_find_dataset(manifest)` resolves them. Workers also have
`ALPHAGO_BASE_DIR=/workspace` set in the docker image.

## Output paths

All outputs land in `experiments/<YYYY-MM-DD_HH-MM>-<slug>/`:
- Edit-target script (`train.py` or `eval_search.py`)
- `results.tsv` — experiment log (untracked)
- `progress.png` — progress plot
- `README.md` — goal and notes
- `report.md` — final findings (overwritten each loop)
